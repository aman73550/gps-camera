To ensure your GPS Map Camera Go app is "unhackable" and only uploads authentic photos captured through the app, here is the consolidated list of every security instruction we have discussed.
🛡️ The "Triple-Lock" Security Framework
This framework ensures that even a tech-savvy user cannot upload a fake or edited image to your server.
1. Private Sandbox Storage (The "Hidden" Vault)
 * Instruction: Do not save photos to the public phone gallery (DCIM or Pictures folder).
 * Technical Implementation: Use getApplicationDocumentsDirectory() in Flutter. This saves files in the app's private internal storage.
 * Security Benefit: System gallery apps cannot see these photos, and users cannot manually "copy-paste" fake photos into this folder unless the phone is rooted.
2. The SQLite "Whitelist" (The Truth Database)
 * Instruction: Maintain a local SQLite database that acts as a "Gatekeeper."
 * Technical Implementation: Every time a photo is clicked, immediately save its File Path, Unique Serial Number, and Timestamp into a local captured_images table.
 * Security Benefit: Before any upload, the app must check if that specific file path exists in the database. If a user somehow "injects" a file into the folder, it won't have a database entry, and the app will block the upload.
3. Hardcoded "No-Picker" Policy
 * Instruction: Strictly prohibit the use of image_picker or any file-selection library.
 * Technical Implementation: The "Files" tab must only read from the internal app directory. There should be no "Select from Gallery" button anywhere in the code.
 * Security Benefit: There is literally no entry point for an external file to enter the app's upload pipeline.
🔒 Upload & Authentication Security
4. Verification Chain (Before Server Upload)
 * Instruction: Implement a multi-step check whenever the "Upload" button is pressed.
 * The Chain:
   * Path Check: Is the file located in the internal AppDocuments folder?
   * Database Check: Does this filename exist in the SQLite Whitelist?
   * Tier Check: Is the user a Guest (max 20) or Logged In (Unlimited)?
   * Metadata Match: Does the Serial Number in the DB match the Serial Number burned onto the image?
 * Security Benefit: This ensures that only "verified" assets reach your server.
5. Manual Trigger Only
 * Instruction: Disable all automatic or background uploads.
 * Logic: The compression (200KB - 300KB) and upload must only trigger when a user manually taps "Upload."
 * Security Benefit: Prevents accidental data leaks and ensures the user is intentional about what is being sent to the server.
🖼️ Anti-Forgery Image Processing
6. Pixel-Level "Burn-In" (Hard-Watermarking)
 * Instruction: Do not rely on EXIF metadata (which is easy to edit). Burn the details directly into the image pixels.
 * Technical Implementation: Use the image package to draw the QR Code, Serial Number, and Geo-details on top of the image data.
 * Security Benefit: If someone tries to crop or edit the photo, the QR code or the 30% opacity background box will look distorted, making it easy to spot a fake.
🚦 User Limitations
7. Guest vs. Registered User Controls
 * Guest Mode: Limited to 20 total uploads. Use SharedPreferences or a specific SQLite counter to track this. Once reached, the "Upload" button should trigger a "Login to Continue" popup.
 * Registered Mode: Use Firebase Authentication. Once logged in, the app removes the upload cap.
8. Exporting (Save to Gallery)
 * Instruction: Provide a "Save to Mobile Gallery" button, but only as a One-Way Export.
 * Security Benefit: Users can move their verified photos to their public gallery for personal use, but they can never move them back from the public gallery into the app for uploading.
